# 🎓 Lecture Builder

> **Многоагентная система автоматизированного создания лекционных материалов вузовского уровня на базе Gemini CLI.**

Система берёт на вход конфигурационный файл с темой, дисциплиной, компетенциями и планом лекции — и на выходе автоматически генерирует полноценный учебный текст: с академическим стилем, нумерованными формулами по ГОСТ, библиографией по ГОСТ, разбором примеров, выводами и методической рецензией.

## ⚡ Быстрый старт

### 1. Установка
```bash
# Установить Gemini CLI
npm install -g @google/gemini-cli

# Клонировать репозиторий
git clone https://github.com/f2re/lecture-builder.git
cd lecture-builder
```

### 2. Заполнить конфигурацию
Отредактируй `input/lecture_config.md` (см. [раздел ниже](#-конфигурация-лекции)).

### 3. Запустить
```bash
# Автоматический режим — весь пайплайн одной командой
gemini build-lecture

# Только литературный анализ
gemini search-literature

# Только рецензия готового черновика
gemini review-lecture

# Написать один конкретный раздел (N — номер вопроса)
gemini write-section N
```

### 4. Результат
Готовая лекция — в `output/lecture_final.md` и **`output/lecture_final.docx`** (ГОСТ).

## 📄 Конвертация в DOCX (ГОСТ 7.32)

В проект интегрирован мощный конвертер `md2docx`, который превращает Markdown в полноценный документ Word с соблюдением требований ГОСТ 7.32.

### Особенности конвертера:
- **Нативные формулы Word**: LaTeX-формулы конвертируются в объекты Office Math (OMML), которые можно редактировать во встроенном редакторе формул Word.
- **Стили ГОСТ**: Автоматическая настройка полей (левое 30мм, правое 15мм), шрифтов (Times New Roman 14pt), межстрочного интервала (18pt) и абзацного отступа (1.25см).
- **Автоматизация**: Скрипт сам создает виртуальное окружение, устанавливает зависимости и настраивает стили через `reference.docx`.
- **Оглавление и нумерация**: Автоматическое создание оглавления и нумерация страниц в нижнем колонтитуле.

### Установка зависимостей
Конвертер требует установленного **pandoc**. Скрипт попытается найти его в системе или загрузить автоматически.

Если pandoc не найден, установите его вручную:
```bash
# Linux (Debian/Ubuntu)
sudo apt install pandoc

# macOS
brew install pandoc

# Windows (через Python обертку, скрипт сделает это сам)
python3 -c "import pypandoc; pypandoc.download_pandoc()"
```

### Запуск
Конвертация запускается **автоматически** в конце полного пайплайна `gemini build-lecture`.

Для ручного запуска:
```bash
# Из корня проекта
bash scripts/md2docx/run_md2docx.sh output/lecture_final.md -o output/lecture_final.docx
```

## 🏗 Архитектура

Система разбита на **10 специализированных агентов**, объединённых в два уровня:

```
┌─────────────────────────────────────────────────────────────┐
│  orchestrator  [Flash]  — автозапуск полного пайплайна      │
└──────────────────────┬──────────────────────────────────────┘
                       │
         ┌─────────────▼──────────────┐
         │  literature-analyst [Flash] │  ← координатор литпайплайна
         │    ├─ lit-searcher  [Flash] │  → output/lit/search_results.json
         │    ├─ lit-fetcher   [Flash] │  → output/lit/extracted_fragments.json
         │    └─ lit-report    [Pro]   │  → bibliography.json + key_concepts.md
         └─────────────┬──────────────┘
                       │
         ┌─────────────▼──────────────┐
         │  query-builder     [Flash] │  → output/queries/query_{N}.md
         └─────────────┬──────────────┘
                       │
         ┌─────────────▼──────────────┐
         │  section-writer ×N  [Pro]  │  → output/sections/section_{N}_*.md
         └─────────────┬──────────────┘
                       │
         ┌─────────────▼──────────────┐
         │  document-assembler [Flash] │  → output/lecture_draft.md
         └─────────────┬──────────────┘
                       │
         ┌─────────────▼──────────────┐
         │  reviewer          [Pro]  │  → output/review_report.md
         └─────────────┬──────────────┘
                       │
         ┌─────────────▼──────────────┐
         │  editor            [Pro]  │  → output/lecture_final.md
         └────────────────────────────┘
```

### Философия модели выбора

Все задачи делятся на два класса:
- **I/O-bound** (поиск, загрузка, сборка) → `gemini-2.5-flash` — в 5–10× быстрее, ~80% дешевле
- **Reasoning-heavy** (синтез, написание, рецензия) → `gemini-2.5-pro` — качество рассуждений имеет значение

Pro-модель задействуется только там, где это реально влияет на качество результата.

## 🤖 Агенты и модели

| Агент | Модель | Роль |
|---|---|---|
| `orchestrator` | Flash | Автоматически запускает весь пайплайн в правильном порядке; обрабатывает ошибки и пропуски |
| `literature-analyst` | Flash | Тонкий координатор: принимает тему → запускает lit-searcher → lit-fetcher → lit-report → собирает результаты |
| `lit-searcher` | Flash | Строит матрицу поиска (RU+EN), запускает `google_web_search`, индексирует локальные файлы из `input/literature/` |
| `lit-fetcher` | Flash | Загружает топ-15 URL через `web_fetch`, извлекает релевантные фрагменты по теме запроса |
| `lit-report` | Pro | Оценивает источники по критериям ФГОС, строит аннотированную библиографию, синтезирует глоссарий |
| `query-builder` | Flash | Расширяет вопросы лекции в полноценные запросы с контекстом, компетенциями и ключевыми понятиями |
| `section-writer` | Pro | Пишет один раздел лекции в научно-педагогическом стиле (1500–2500 слов); нумерует формулы по ГОСТ |
| `document-assembler` | Flash | Собирает разделы в единый черновик; проверяет naming convention, связность и полноту |
| `reviewer` | Pro | Методическая рецензия: соответствие ФГОС, педагогическая логика, полнота компетенций, актуальность источников |
| `editor` | Pro | Финальная правка: стиль, единство терминологии, ссылки на формулы, грамматика; ведёт `edit_log.md` |

## 🧠 Скиллы (общие знания)

Скиллы — переиспользуемые файлы знаний, которые подключаются к агентам через `@../skills/`. Загружаются только тогда, когда нужны, не расходуя токены впустую.

| Файл | Содержание |
|---|---|
| `search-patterns.md` | Шаблоны поисковых запросов для КиберЛенинки, eLibrary, arXiv, Google Scholar, РИНЦ |
| `gost-citation.md` | Правила оформления библиографии по ГОСТ Р 7.0.5-2008 |
| `fgos-standards.md` | Структура компетенций ФГОС 3++: ОПК, ПК, индикаторы, форматы проверки |
| `formula-numbering.md` | Нумерация формул по ГОСТ 2.105-2019: формат `(N.M)`, `\tag{}` для pandoc→DOCX, счётчик `eq_counter`, расшифровка символов |

## ⚙️ Конфигурация лекции

Основной входной файл: **`input/lecture_config.md`**. Формат — YAML-блок:

```yaml
topic: "Барическое поле и атмосферное давление"
discipline: "Синоптическая метеорология"
specialty: "05.03.04 Гидрометеорология"
course: "3 курс, 6 семестр, Лекция 5"
hours: 2
fgos_version: "ФГОС 3++"
competencies:
  - "ОПК-3: способность применять математические методы..."
  - "ПК-1: готовность к синоптическому анализу карт..."
audience_level: "бакалавры 3 курса"
questions:
  - "1. Барическое поле и его характеристики"
  - "2. Горизонтальный барический градиент"
  - "3. Основное уравнение статики атмосферы"
language: ru
formulas_required: true
```

### Поля конфигурации

| Поле | Обязательно | Описание |
|---|---|---|
| `topic` | ✅ | Название лекции — используется во всех агентах |
| `discipline` | ✅ | Название дисциплины (входит в шапку лекции) |
| `specialty` | ✅ | Код и название специальности по ФГОС |
| `course` | ✅ | Курс, семестр, номер лекции |
| `hours` | ✅ | Кол-во академических часов (влияет на объём: 2 ч → ≤8000 слов) |
| `fgos_version` | ✅ | Версия ФГОС (`ФГОС 3++` по умолчанию) |
| `competencies` | ✅ | Список компетенций с кодами — reviewer проверяет их покрытие |
| `questions` | ✅ | Пронумерованные вопросы плана лекции |
| `audience_level` | — | Уровень аудитории (влияет на стиль изложения) |
| `formulas_required` | — | `true` — section-writer активирует нумерацию формул |
| `language` | — | Язык вывода (по умолчанию `ru`) |

Дополнительно можно положить в `input/literature/` локальные PDF-учебники — `lit-searcher` проиндексирует их.

## 🔄 Режимы работы

### Автоматический режим (рекомендуется)

Запускает весь пайплайн от конфига до финального файла:

```bash
gemini build-lecture
```

Оркестратор сам определяет, какие шаги уже выполнены (по наличию файлов в `output/`), и пропускает их. Это позволяет **перезапускать с любого этапа** без потери предыдущих результатов.

### Частичные команды

```bash
# Только литературный анализ (3 этапа: поиск → загрузка → отчёт)
gemini search-literature

# Написать конкретный раздел (N = номер вопроса из плана)
gemini write-section 3

# Рецензия уже написанного черновика
gemini review-lecture
```

### Ручной режим (для отладки)

Каждый агент вызывается напрямую через `@`:

```
@agents/literature-analyst.md       # шаг 1: литература
@agents/query-builder.md            # шаг 2: расширение вопросов
@agents/section-writer.md           # шаг 3: написание раздела (N указать явно)
@agents/document-assembler.md       # шаг 4: сборка черновика
@agents/reviewer.md                 # шаг 5: рецензия
@agents/editor.md                   # шаг 6: редактура
```

> **Важно:** При ручном вызове `section-writer` нужно передать номер вопроса N явно в сообщении, например: `«Напиши раздел 2»`.

## 📐 Стандарты и требования

### ФГОС 3++
- `query-builder` разворачивает каждый вопрос с учётом компетенций из конфига
- `reviewer` явно проверяет, покрыт ли каждый код компетенции (ОПК-N, ПК-N) содержанием лекции
- `lit-report` оценивает источники по критериям актуальности ФГОС (не старше 5–10 лет)

### ГОСТ Р 7.0.5-2008 (библиография)
- Скилл `gost-citation.md` задаёт правила оформления для `lit-report`
- Формат инлайн-ссылок: `[Иванов, 2021, с. 45]`
- Все ref_id в тексте должны присутствовать в `bibliography.json`

### ГОСТ 2.105-2019 (формулы)
- Скилл `formula-numbering.md` задаёт правила для `section-writer`
- Нумерация: `(N.M)` — N = номер лекции, M = счётчик в лекции
- Формат в Markdown: `$$ формула \tag{N.M} $$` → pandoc конвертирует в OMML (Word)
- Счётчик `eq_counter` сбрасывается в 1 для каждой новой лекции
- После каждой блочной формулы — обязательная расшифровка символов
- Ссылки в тексте: «из выражения (3.1) следует...»

## ⚠️ Нюансы и ограничения

### Контекст и токены
- `section-writer` запускается отдельно для каждого вопроса (раздела) — это **намеренно**, чтобы не переполнять контекст Pro-модели при большом числе вопросов
- Скиллы загружаются только в нужных агентах: `formula-numbering.md` — только в `section-writer`, `gost-citation.md` — только в `lit-report`
- Промпты воркфлоу на **английском** (у агентов lit-*) — это снижает расход токенов на ~20%

### Промежуточные файлы
- Оркестратор проверяет наличие файлов в `output/` перед запуском каждого шага
- Если `search_results.json` уже есть — `lit-searcher` пропускается
- Это позволяет **дозапускать пайплайн** после ошибки без повтора дорогих шагов

### Ручное вмешательство
- Перед запуском `document-assembler` можно вручную отредактировать любой `section_{N}_*.md`
- `reviewer` читает черновик и пишет замечания — `editor` использует их как инструкцию
- Финальный `lecture_final.md` всегда содержит оригинальный `lecture_draft.md` как точку отката

### Локальная литература
- PDF-файлы в `input/literature/` индексируются `lit-searcher` через `grep_search`
- Поддерживаются: `.md`, `.txt`, `.pdf` (если Gemini CLI умеет их читать)
- При отсутствии локальных файлов — только веб-поиск

### Ограничения
- Система не генерирует иллюстрации сама — только промпты в `image_prompts.md`
- Конвертация в DOCX требует pandoc: `pandoc output/lecture_final.md -o lecture.docx --mathml`
- Формулы в DOCX корректны при использовании `\tag{}` (конвертируется в OMML через pandoc)
