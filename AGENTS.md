## 🎯 Project Purpose

**Lecture Builder** is a multi-agent system for generating university-level lecture materials in Russian using Gemini CLI.

The system reads `input/lecture_config.md` and produces:

- `output/lecture_final.md` — final lecture text.
- `output/lecture_final.docx` — ГОСТ DOCX generated by `scripts/md2docx/run_md2docx.sh`.
- `output/bibliography.json` — annotated bibliography.
- `output/key_concepts.md` — glossary of key concepts.
- `output/review_report.md` — methodological review.
- `output/edit_log.md`, `output/image_prompts.md`, `output/figures_index.json`.

1. Read this `AGENTS.md`.
2. Read `.codex/agents/README.md` and select the active profile.

5. Read the matching workflow in `.gemini/workflows/<profile>.md` if present.
6. Read relevant skills from `.gemini/skills/` only when the task needs them.
7. Inspect existing input/output files before creating or overwriting generated artifacts.
8. Produce a plan before editing repository files.

## 🔄 Mandatory 4-Phase Lifecycle

### Phase 1 — PLAN `[xhigh reasoning]`

**Do not write files.**

- Identify the active profile.
- Read the relevant Gemini agent and workflow.
- Inspect existing `input/`, `output/`, `.gemini/`, `scripts/`, and generated artifacts affected by the task.
- Write a 5–10 step plan.
- Define verification commands and expected output files.
- State whether the task modifies source prompts, workflows, generated lecture content, or conversion scripts.

### Phase 2 — IMPLEMENT `[high reasoning]`

- Follow the Phase 1 plan.
- Keep generated artifacts in `output/` and source configuration in `input/`.
- Keep agent definitions in `.gemini/agents/` or `.codex/agents/`.
- Keep workflow instructions in `.gemini/workflows/`.
- Keep reusable domain rules in `.gemini/skills/`.
- Do not overwrite user-provided input/literature unless explicitly asked.

### Phase 3 — VERIFY `[xhigh reasoning]`

- Verify all expected output files exist and are non-empty.
- For Markdown output, check headings, bibliography references, formula numbering, and required sections.
- For JSON output, validate parseability.
- For DOCX output, run the conversion script when applicable.
- If a command requires network/Gemini/Pandoc and it is unavailable, record an explicit blocker.

### Phase 4 — FIX

- Fix only concrete errors found in Phase 3.
- Repeat Phase 3 after each fix.
- Do not guess or mask failures.

## 🧭 Codex Agent Router

| Profile | Use for | Gemini source | Workflow | Primary outputs |
|---|---|---|---|---|
| `orchestrator` | full pipeline coordination and status verification | `.gemini/agents/orchestrator.md` | `.gemini/workflows/orchestrator.md` | final status summary |
| `literature-analyst` | literature pipeline coordination | `.gemini/agents/literature-analyst.md` | `.gemini/workflows/literature-analyst.md` | bibliography, map, concepts |
| `lit-searcher` | search query matrix and raw source discovery | `.gemini/agents/lit-searcher.md` | `.gemini/workflows/lit-searcher.md` | `output/lit/search_results.json` |
| `lit-fetcher` | fetch URLs and extract relevant fragments | `.gemini/agents/lit-fetcher.md` | `.gemini/workflows/lit-fetcher.md` | `output/lit/extracted_fragments.json` |
| `lit-report` | annotated bibliography and concept synthesis | `.gemini/agents/lit-report.md` | `.gemini/workflows/lit-report.md` | `bibliography.json`, `key_concepts.md` |
| `query-builder` | expand lecture questions into section prompts | `.gemini/agents/query-builder.md` | `.gemini/workflows/query-builder.md` | `output/queries/query_N.md` |
| `section-writer` | write one lecture section | `.gemini/agents/section-writer.md` | `.gemini/workflows/section-writer.md` | `output/sections/section_N_*.md` |
| `document-assembler` | assemble draft from sections | `.gemini/agents/document-assembler.md` | `.gemini/workflows/document-assembler.md` | `output/lecture_draft.md` |
| `reviewer` | methodological review | `.gemini/agents/reviewer.md` | `.gemini/workflows/reviewer.md` | `output/review_report.md` |
| `editor` | final edit and DOCX-ready Markdown | `.gemini/agents/editor.md` | `.gemini/workflows/editor.md` | final lecture and edit artifacts |

### Routing Rules

- The orchestrator coordinates only; it must not write lecture content, bibliography, sections, reviews, or final edits.
- `literature-analyst` coordinates the literature sub-pipeline and delegates to `lit-searcher`, `lit-fetcher`, and `lit-report`.
- `section-writer` is called once per question/query file, not once for the full lecture.
- `document-assembler` assembles existing sections; it must not invent missing sections.
- `reviewer` evaluates; `editor` applies final edits.
- DOCX generation belongs to the final pipeline step and must use `scripts/md2docx/run_md2docx.sh`.

## 📐 Academic Standards

### ФГОС 3++

- Queries and sections must reflect the configured competencies.
- Reviewer must explicitly check coverage of each competency code.

### ГОСТ Р 7.0.5-2008

- Bibliography formatting and inline references follow `.gemini/skills/gost-citation.md`.
- Inline references use the form `[Автор, год, с. X]` when source details are available.
- All references in the lecture must be present in `bibliography.json`.

### ГОСТ 2.105-2019 formulas

- Formula rules follow `.gemini/skills/formula-numbering.md`.
- Block formulas use `\tag{N.M}` for Pandoc/Word conversion.
- Every block formula needs symbol explanations and text references such as `(3.1)`.

## 🔧 Verification Commands

Use the most specific checks for the profile. Defaults:

```bash
# Check required config exists
python3 - <<'PY'
from pathlib import Path
p = Path('input/lecture_config.md')
assert p.exists() and p.read_text(encoding='utf-8').strip(), 'input/lecture_config.md is missing or empty'
PY

# Validate JSON artifacts when present
python3 - <<'PY'
import json
from pathlib import Path
for p in Path('output').rglob('*.json'):
    json.loads(p.read_text(encoding='utf-8'))
    print(f'OK {p}')
PY

# DOCX conversion when final Markdown exists
test -f output/lecture_final.md && bash scripts/md2docx/run_md2docx.sh output/lecture_final.md -o output/lecture_final.docx || true
```

If Pandoc, Gemini CLI, browser/network, or external fetch tools are unavailable, record the exact blocker instead of reporting success.

## ✅ MUST

- Follow the 4-phase lifecycle.
- Read relevant Gemini workflow before changing or executing that part of the pipeline.
- Preserve the I/O contracts between agents.
- Keep source prompts/workflows separate from generated `output/` artifacts.

- Use ГОСТ and ФГОС skills when relevant.

## ❌ MUST NOT

- Let orchestrator generate specialist content itself.
- Invent bibliography entries or cite sources not present in `bibliography.json`.
- Overwrite `input/lecture_config.md`, `input/existing_refs.md`, or local literature without explicit instruction.
- Put generated lecture artifacts outside `output/`.
- Put agent source files outside `.gemini/agents/` or `.codex/agents/`.
- Delete intermediate artifacts just to make the pipeline pass.
- Hide network/Gemini/Pandoc failures.
- Commit secrets, API keys, cookies, tokens, or private source PDFs that should remain local.
